# Code Review Process ## Mission Ensure all code changes are reviewed by an independent party before merging to maintain quality, security, and consistency. ## Review Roles ### Primary Reviewer - **Role:** Independent developer not involved in the original implementation - **Responsibilities:** - Review code for bugs, security issues, and best practices - Verify tests cover edge cases - Ensure code follows project conventions - Check for performance implications - Validate that the implementation matches requirements ### When Review is Required - [x] All new features - [x] All bug fixes (especially critical ones) - [x] Performance optimizations - [x] Security-related changes - [x] Breaking changes or API modifications - [ ] Trivial changes (typo fixes, documentation only) ## Review Checklist ### Before Submitting for Review - [ ] Code compiles without errors or warnings - [ ] All tests pass - [ ] Code is formatted according to project standards - [ ] Commit messages are clear and descriptive - [ ] Documentation updated (if needed) - [ ] Self-review completed (developer reviewed their own code first) ### Reviewer Responsibilities - [ ] **Functionality:** Does the code do what it's supposed to? - [ ] **Security:** No vulnerabilities introduced (SQL injection, XSS, etc.) - [ ] **Performance:** No obvious performance issues or memory leaks - [ ] **Testing:** Adequate test coverage for the changes - [ ] **Readability:** Code is clear and well-commented - [ ] **Consistency:** Follows existing code style and patterns - [ ] **Dependencies:** No unnecessary or risky new dependencies ### Android-Specific Checks - [ ] No hardcoded strings (use string resources) - [ ] No hardcoded dimensions (use dimens.xml) - [ ] Proper lifecycle management (no memory leaks) - [ ] Async operations on background threads - [ ] UI updates on main thread - [ ] Permissions handled correctly - [ ] No sensitive data in logs - [ ] APK builds successfully in release mode ## Review Workflow 1. **Developer completes feature/fix** - Works on feature branch - Writes tests - Self-reviews 2. **Submit for review** - Push branch to GitHub - Create Pull Request - Tag independent reviewer - Add description of changes and testing done 3. **Reviewer evaluates** - Reviews code within 24 hours - Leaves constructive feedback - Approves or requests changes 4. **Address feedback** - Developer makes requested changes - Re-submits for review if needed 5. **Merge** -
